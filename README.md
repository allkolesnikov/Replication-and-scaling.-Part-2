## Задание 1

### 1. Активный master-сервер и пассивный slave-сервер

При такой схеме основной сервер выполняет все операции чтения и записи, а slave-сервер получает копию данных через репликацию и находится в резерве.
Преимущества:
- Главное преимущество — отказоустойчивость. Если master-сервер выходит из строя, slave можно быстро перевести в режим master и продолжить работу системы.
- Также такая схема удобна для резервного копирования: бэкапы можно снимать со slave-сервера, не нагружая основной master.
- Ещё один плюс — защита от простоев при обслуживании. Например, master можно обновлять или ремонтировать после переключения нагрузки на slave.
- Минус схемы в том, что производительность почти не растёт, потому что slave обычно не участвует активно в обработке пользовательских запросов.

### 2. Master-сервер и несколько slave-серверов

В этой схеме master-сервер принимает все операции записи, а несколько slave-серверов обслуживают операции чтения.
Преимущества:
- Основное преимущество — масштабирование чтения. Если приложение часто читает данные, нагрузку можно распределить между несколькими slave-серверами.
- Также повышается отказоустойчивость: при выходе из строя одного slave остальные продолжают работать.
- Slave-серверы можно использовать под разные задачи: один для пользовательских запросов, второй для аналитики, третий для резервного копирования.
- Такая архитектура хорошо подходит для интернет-магазинов, библиотек, маркетплейсов и других систем, где чтений намного больше, чем записей.


## Задание 2

База данных состоит из трёх таблиц:
```
users
- id
- name
- email
- region
- created_at

books
- id
- title
- author
- genre
- price
- shop_id

shops
- id
- name
- city
- region
```

### Горизонтальный шардинг

- Горизонтальный шардинг — это разделение строк одной таблицы между разными базами данных.

Например, пользователей и магазины можно делить по региону:
- DB_SHARD_1 — регион EU
- DB_SHARD_2 — регион US
- DB_SHARD_3 — регион ASIA

Пример:
```
users с region = 'EU'    → DB_SHARD_1
users с region = 'US'    → DB_SHARD_2
users с region = 'ASIA'  → DB_SHARD_3

shops с region = 'EU'    → DB_SHARD_1
shops с region = 'US'    → DB_SHARD_2
shops с region = 'ASIA'  → DB_SHARD_3
```
Таблицу books логично размещать рядом с магазином, которому принадлежит книга:
- books.shop_id → shops.id → region → нужный shard

То есть книги европейских магазинов будут лежать в европейском шарде, книги американских магазинов — в американском.

### Вертикальный шардинг

- Вертикальный шардинг — это разделение таблиц или групп столбцов по разным базам данных.

Например:
- DB_USERS — пользователи
- DB_CATALOG — книги
- DB_SHOPS — магазины

Также можно отделить редко используемые или тяжёлые данные:
```
DB_USERS_MAIN
- id
- name
- email

DB_USERS_PROFILE
- user_id
- address
- phone
- preferences
```
Для книг:
```
DB_BOOKS_MAIN
- id
- title
- author
- price
- shop_id

DB_BOOKS_EXTRA
- book_id
- description
- cover_image
- metadata
```
Так уменьшается нагрузка на основные таблицы, потому что часто используемые данные находятся отдельно от тяжёлых и редко запрашиваемых.

### Режимы работы серверов
Master-сервер:
- принимает INSERT, UPDATE, DELETE;
- является основным источником данных;
- передаёт изменения на slave-серверы.

Slave-серверы:
- принимают SELECT-запросы;
- используются для чтения;
- получают данные через репликацию от master;
- могут использоваться для резервного копирования и аналитики.

Shard-серверы:
- каждый хранит только свою часть данных;
- например, отдельный регион или диапазон id;
- внутри каждого shard может быть свой master и свои slave.

Итоговая оптимальная архитектура:
1. Горизонтально делить данные по region.
2. Таблицы users, shops и books хранить в одном региональном shard.
3. Записи направлять только на master.
4. Чтение распределять между slave-серверами.
5. Тяжёлые поля вроде описаний, изображений и настроек вынести вертикально в отдельные таблицы или базы.
